## 实现计划

### 1. 重构目标
将 `collections.py` 中 `upsert_collection_endpoint` 函数的业务逻辑提取到 `collection_service.py` 中，遵循分层架构设计规范。

### 2. 核心业务逻辑分析
原函数主要包含以下业务逻辑：
- 验证请求数据
- 检查收藏是否存在
- 检查条目是否已存在
- 日期格式验证和冲突检查
- 创建/更新 Subject 记录
- 创建/更新 Collection 记录
- 清除缓存

### 3. 重构方案

#### 3.1 在 `collection_service.py` 中添加新函数
- **函数名**：`upsert_collection`
- **参数**：
  - `db: AsyncSession`：数据库会话
  - `user_id: int`：当前用户ID
  - `sid: Optional[int]`：条目ID
  - `data: CollectionUpsertRequest`：请求数据
- **返回值**：`Collection` 对象
- **功能**：封装所有收藏更新/添加的业务逻辑

#### 3.2 函数实现步骤
1. **验证请求数据**：检查 `sid` 和 `data` 的合法性
2. **日期格式验证**：使用 `validate_date_format` 函数验证日期格式
3. **日期冲突检查**：使用 `check_date_conflict` 函数检查日期冲突
4. **处理 Subject 数据**：
   - 如果 `sid` 不存在，使用 `SubjectRepo.save` 创建新 Subject
   - 如果 `sid` 存在，使用 `SubjectRepo.save` 更新现有 Subject
5. **处理 Collection 数据**：
   - 使用 `CollectionRepo.save` 创建/更新 Collection 记录
6. **返回结果**：返回处理后的 Collection 对象

#### 3.3 更新 `collections.py`
- 导入新的 `upsert_collection` 函数
- 简化 `upsert_collection_endpoint` 函数，只保留路由和缓存清除逻辑
- 调用 `collection_service.upsert_collection` 处理业务逻辑

### 4. 代码复用与一致性
- **复用现有模式**：遵循 `SubjectRepo` 和 `CollectionRepo` 的设计模式
- **保持逻辑顺序**：先更新 Subject，后更新 Collection（参考 `bangumi_service.py`）
- **使用相同接口**：使用 `SubjectRepo.save` 和 `CollectionRepo.save` 进行数据操作

### 5. 预期效果
- API 功能不受影响
- 代码结构清晰，符合分层架构
- 业务逻辑与路由逻辑分离
- 便于维护和扩展

### 6. 技术细节
- 保留原有日期验证和冲突检查逻辑
- 保持原有错误处理机制
- 确保缓存清除逻辑正常执行
- 使用与现有代码一致的命名规范和代码风格

- 验证创建新收藏功能正常
- 验证日期验证和冲突检查功能正常
- 验证错误处理机制正常

# 解决时间处理错误问题

## 1. 错误分析

### 1.1 错误信息
```
invalid input for query argument $7: datetime.datetime(2025, 12, 30, 11, 17, ... (can't subtract offset-naive and offset-aware datetimes)
```

### 1.2 错误类型
- 类型：`asyncpg.exceptions.DataError`
- 根本原因：无法在带时区的datetime对象和不带时区的datetime对象之间进行减法运算

### 1.3 问题根源
1. **User模型定义**：`created_at`和`updated_at`字段使用`utc_now()`函数作为默认值，该函数返回**带时区信息**的datetime对象（offset-aware）
2. **数据库字段类型**：数据库中的字段类型为`TIMESTAMP WITHOUT TIME ZONE`，**不支持时区信息**
3. **SQLAlchemy转换**：当尝试将带时区的datetime对象插入到不带时区的数据库字段时，SQLAlchemy会尝试进行转换，导致类型不兼容错误

## 2. 解决方案

### 2.1 修复方案
修改`user.py`文件中的`utc_now()`函数，使其返回**不带时区信息**的UTC时间：

```python
def utc_now():
    """获取当前 UTC 时间，返回不带时区信息的datetime对象"""
    return datetime.utcnow()
```

### 2.2 修复原理
- `datetime.utcnow()`返回不带时区信息的UTC时间（offset-naive）
- 与数据库字段类型`TIMESTAMP WITHOUT TIME ZONE`兼容
- 避免了SQLAlchemy进行复杂的时区转换

## 3. 修复步骤

1. 修改`user.py`文件中的`utc_now()`函数
2. 重启后端服务
3. 测试登录接口，验证问题是否解决

## 4. 测试方案

- 测试场景1：新用户注册
- 测试场景2：现有用户信息更新
- 测试场景3：多次调用登录接口

验证所有场景下都能成功返回JWT token和用户信息，没有时间处理相关错误。

## 5. 预期效果

- 登录接口能够正常处理用户注册和信息更新
- 数据库能够正确存储用户的创建时间和更新时间
- 不再出现"can't subtract offset-naive and offset-aware datetimes"错误

## 6. 风险评估

- **低风险**：修改仅涉及时间生成方式，不影响其他业务逻辑
- **向后兼容**：现有数据不会受到影响，新数据将使用正确的时间格式
- **性能提升**：减少了不必要的时区转换，提高了数据库操作效率

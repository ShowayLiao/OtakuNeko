# 问题分析与修复计划

## 1. 问题分析

### 1.1 错误现象
- 前端调用 `getUserCollections` 函数时出现错误
- 后端 Terminal 显示 `sqlalchemy.exc.ProgrammingError` 错误，具体为 `operator does not exist: integer = character varying`
- 错误发生在执行 SQL 查询时：`SELECT users.id, users.username, ... WHERE users.id = $1::VARCHAR`

### 1.2 根本原因
- 在后端 `auth.py` 文件中，创建 JWT 令牌时，`sub` 字段被转换为了字符串类型：
  ```python
  token = create_access_token(data={"sub": str(new_user.id), "username": new_user.username})
  ```

- 然而，在 `deps.py` 文件中，`get_current_user` 函数直接将 `sub` 字段作为整数类型使用：
  ```python
  user_id: Optional[int] = payload.get("sub")
  result = await db.execute(
      select(User).where(User.id == user_id)
  )
  ```

- 当 SQLAlchemy 执行查询时，它发现 `user_id` 是字符串类型，而 `User.id` 是整数类型，因此抛出了类型不匹配的错误。

## 2. 修复方案

### 2.1 修复位置
- 文件：`e:\HACCI\Documents\tools\OtakuNeko\backend\app\api\deps.py`
- 函数：`get_current_user`

### 2.2 修复方法
在 `get_current_user` 函数中，将从 JWT 令牌中获取的 `sub` 字段显式转换为整数类型，然后再用于 SQL 查询。

### 2.3 修复代码
将以下代码：
```python
user_id: Optional[int] = payload.get("sub")
if user_id is None:
    raise credentials_exception
```

修改为：
```python
user_id_str: Optional[str] = payload.get("sub")
if user_id_str is None:
    raise credentials_exception

# 将字符串类型的 user_id 转换为整数类型
try:
    user_id = int(user_id_str)
except ValueError:
    raise credentials_exception
```

该问题是由于 JWT 令牌中 `sub` 字段的类型与数据库查询时的类型不匹配导致的。通过在 `get_current_user` 函数中添加显式的类型转换，可以解决这个问题。修复后，所有需要认证的 API 端点都能正常工作。

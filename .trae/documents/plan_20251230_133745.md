# API端点测试计划

## 1. 测试环境准备
- 确保后端服务已启动，运行在正确的端口（默认8000）
- 使用Postman或curl等工具进行API测试
- 准备测试数据，包括用户名、密码（如果需要）、JWT令牌等

## 2. 测试顺序
按照API功能模块顺序进行测试：
1. 认证相关API（Auth）
2. 用户相关API（Users）
3. 仪表板相关API（Dashboard）
4. 条目相关API（Subjects）
5. 收藏相关API（Collections）

## 3. 测试用例设计

### 3.1 认证相关API

#### 3.1.1 POST /auth/login
- **功能**：用户登录/注册/信息更新综合接口
- **测试用例1**：新用户注册
  - 请求数据：`{"username": "testuser1", "nickname": "测试用户1", "sign": "测试签名"}`
  - 预期结果：返回200状态码，包含access_token、token_type和user信息
  - 验证点：
    - 响应状态码为200
    - 响应格式符合LoginResponse模型
    - user信息包含正确的username、nickname等

- **测试用例2**：现有用户登录
  - 请求数据：`{"username": "testuser1"}`
  - 预期结果：返回200状态码，包含access_token、token_type和user信息
  - 验证点：
    - 响应状态码为200
    - 响应格式符合LoginResponse模型
    - user信息包含正确的username

- **测试用例3**：现有用户信息更新
  - 请求数据：`{"username": "testuser1", "nickname": "更新后的昵称", "sign": "更新后的签名"}`
  - 预期结果：返回200状态码，包含access_token、token_type和更新后的user信息
  - 验证点：
    - 响应状态码为200
    - 响应格式符合LoginResponse模型
    - user信息包含更新后的nickname和sign

### 3.2 用户相关API

#### 3.2.1 GET /users/me
- **功能**：获取当前登录用户信息
- **测试用例1**：已认证用户获取信息
  - 请求头：`Authorization: Bearer <token>`
  - 预期结果：返回200状态码，包含当前用户的完整信息
  - 验证点：
    - 响应状态码为200
    - 响应包含正确的用户信息
    - 响应格式符合文档要求

### 3.3 仪表板相关API

#### 3.3.1 GET /dashboard/stats
- **功能**：获取用户的仪表板统计数据
- **测试用例1**：已认证用户获取统计数据
  - 请求头：`Authorization: Bearer <token>`
  - 预期结果：返回200状态码，包含各分类（动画/书籍/游戏/音乐/三次元）的收藏数量
  - 验证点：
    - 响应状态码为200
    - 响应格式符合DashboardStats模型
    - 包含anime、book、game、music、real等字段

### 3.4 条目相关API

#### 3.4.1 GET /subjects
- **功能**：搜索条目
- **测试用例1**：使用关键词搜索
  - 请求头：`Authorization: Bearer <token>`
  - 请求参数：`q=测试&source=mixed&type=2&limit=10`
  - 预期结果：返回200状态码，包含搜索结果列表
  - 验证点：
    - 响应状态码为200
    - 响应格式符合SubjectRead模型列表
    - 结果数量不超过10条

#### 3.4.2 GET /subjects/{subject_id}
- **功能**：获取单个条目详情
- **测试用例1**：获取存在的条目详情
  - 请求头：`Authorization: Bearer <token>`
  - 请求参数：`refresh=false`
  - 预期结果：返回200状态码，包含条目详细信息
  - 验证点：
    - 响应状态码为200
    - 响应格式符合SubjectRead模型
    - 包含正确的条目信息

- **测试用例2**：获取不存在的条目详情
  - 请求头：`Authorization: Bearer <token>`
  - 请求参数：`subject_id=999999`
  - 预期结果：返回404状态码
  - 验证点：
    - 响应状态码为404
    - 响应包含错误信息

#### 3.4.3 PUT /subjects/{subject_id}
- **功能**：修改条目信息
- **测试用例1**：修改存在的条目信息
  - 请求头：`Authorization: Bearer <token>`
  - 请求数据：`{"name": "修改后的名称", "score": 9.5}`
  - 预期结果：返回200状态码，包含更新后的条目信息
  - 验证点：
    - 响应状态码为200
    - 响应格式符合SubjectRead模型
    - 条目信息已更新

### 3.5 收藏相关API

#### 3.5.1 GET /collections
- **功能**：获取用户的收藏列表
- **测试用例1**：获取所有收藏
  - 请求头：`Authorization: Bearer <token>`
  - 请求参数：`limit=20&offset=0`
  - 预期结果：返回200状态码，包含收藏列表和总数
  - 验证点：
    - 响应状态码为200
    - 响应格式符合CollectionList模型
    - 包含total和items字段

- **测试用例2**：按类型筛选收藏
  - 请求头：`Authorization: Bearer <token>`
  - 请求参数：`subject_type=2&limit=20&offset=0`
  - 预期结果：返回200状态码，包含指定类型的收藏列表
  - 验证点：
    - 响应状态码为200
    - 响应格式符合CollectionList模型
    - items中的条目类型符合筛选条件

#### 3.5.2 PUT /collections/{sid}
- **功能**：更新或添加收藏
- **测试用例1**：添加新收藏
  - 请求头：`Authorization: Bearer <token>`
  - 请求数据：`{"collection": {"status": 2, "rate": 9, "comment": "很好看的动画"}, "subject": {"name": "测试动画", "type": 2, "cover_url": "https://example.com/cover.jpg", "release_date": "2023-01-01", "tags": ["动画", "热血"]}}`
  - 预期结果：返回200状态码，包含收藏信息
  - 验证点：
    - 响应状态码为200
    - 响应格式符合CollectionRead模型
    - 包含正确的收藏信息

- **测试用例2**：更新现有收藏
  - 请求头：`Authorization: Bearer <token>`
  - 请求数据：`{"collection": {"status": 3, "rate": 8, "comment": "更新后的评论"}}`
  - 预期结果：返回200状态码，包含更新后的收藏信息
  - 验证点：
    - 响应状态码为200
    - 响应格式符合CollectionRead模型
    - 收藏信息已更新

#### 3.5.3 POST /collections/sync
- **功能**：同步收藏（支持BGM和豆瓣）
- **测试用例1**：同步BGM收藏
  - 请求头：`Authorization: Bearer <token>`
  - 请求数据：`{"source": "bgm", "subject_type": 2}`
  - 预期结果：返回200状态码，包含同步结果
  - 验证点：
    - 响应状态码为200
    - 响应包含sync_count字段
    - 响应消息包含成功信息

- 所有API端点均返回200状态码（除了测试不存在资源的情况）

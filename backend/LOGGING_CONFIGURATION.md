# 日志配置文档

## 1. 日志系统概述

本后端系统采用了完整的日志记录方案，实现了日志的系统化管理与存储，确保系统运行状态的可追溯性和问题的快速定位。

## 2. 日志配置详情

### 2.1 日志级别
系统支持以下日志级别（从高到低）：
- **ERROR**: 错误信息，如系统异常、数据库连接失败等
- **WARN**: 警告信息，如参数验证失败、缓存过期等
- **INFO**: 信息性消息，如系统启动、重要操作执行等
- **DEBUG**: 调试信息，如详细的函数调用、变量值等

### 2.2 日志格式
日志格式包含以下信息：
```
2026-01-15 12:00:00 - INFO - app.main - req-12345 - Redis cache initialized successfully
```
- **时间戳**: 日志记录的时间，格式为 `YYYY-MM-DD HH:MM:SS`
- **日志级别**: ERROR、WARN、INFO、DEBUG 之一
- **模块名称**: 产生日志的模块，如 `app.main`、`app.repositories.subject_repo`
- **请求ID**: 用于追踪单个请求的完整调用链，默认为 `-`
- **具体日志内容**: 日志的详细信息

### 2.3 日志存储
- **日志目录**: `backend/logs/`
- **文件命名**: 
  - 普通日志: `app.log`
  - 错误日志: `error.log`
- **轮转策略**:
  - **按日期轮转**: 每天生成新的日志文件
  - **按大小轮转**: 单个日志文件达到 100MB 时自动轮转
  - **保留策略**: 默认保留 30 天的日志记录

### 2.4 性能优化
- **异步写入**: 使用 `ConcurrentRotatingFileHandler` 实现异步日志写入，减少对系统性能的影响
- **缓冲区**: 配置了适当的缓冲区大小，减少 I/O 操作
- **日志级别控制**: 生产环境可设置较高的日志级别，减少日志量

### 2.5 日志清理
- **自动清理**: 超过保留期的日志文件会被自动清理
- **归档机制**: 超过保留期的日志文件会先归档为压缩文件，然后在一定时间后删除

## 3. 日志配置修改

### 3.1 配置项
日志配置项位于 `app/core/config.py` 文件中的 `Settings` 类：
- `LOG_LEVEL`: 日志级别，默认为 "INFO"
- `LOG_DIR`: 日志文件目录，默认为 "logs"
- `LOG_MAX_BYTES`: 单个日志文件最大字节数，默认为 104857600 (100MB)
- `LOG_BACKUP_COUNT`: 保留的日志文件数量，默认为 30
- `LOG_FORMAT`: 日志格式，包含时间戳、日志级别、模块名称、请求ID、具体日志内容
- `LOG_DATE_FORMAT`: 日期格式，默认为 "%Y-%m-%d %H:%M:%S"

### 3.2 环境变量配置
可以通过环境变量覆盖默认配置：
```bash
# .env 文件示例
LOG_LEVEL=DEBUG
LOG_DIR=custom_logs
LOG_BACKUP_COUNT=15
```

## 4. 日志使用方法

### 4.1 在代码中使用日志
```python
from app.core.logging import get_logger

# 获取日志记录器
logger = get_logger(__name__)

# 记录不同级别的日志
logger.debug("这是一条调试信息")
logger.info("这是一条信息")
logger.warning("这是一条警告")
logger.error("这是一条错误信息")
logger.critical("这是一条严重错误信息")
```

### 4.2 请求ID追踪
在处理 HTTP 请求时，可以设置请求ID以追踪整个请求链路：
```python
from app.core.logging import set_request_id, clear_request_id

async def some_handler(request: Request):
    # 生成并设置请求ID
    request_id = str(uuid.uuid4())
    set_request_id(request_id)
    
    try:
        # 处理请求
        logger.info("开始处理请求")
        # ...
    finally:
        # 清除请求ID
        clear_request_id()
```

## 5. 日志查看方法

### 5.1 实时查看
```bash
# 查看普通日志
tail -f logs/app.log

# 查看错误日志
tail -f logs/error.log
```

### 5.2 历史日志查看
日志文件会按日期轮转，例如：
- `app.log` - 当前日志
- `app.log.2026-01-14` - 2026年1月14日的日志
- `app.log.2026-01-13` - 2026年1月13日的日志

可以使用 `cat`、`less` 或 `grep` 等命令查看历史日志：
```bash
# 查看特定日期的日志
cat logs/app.log.2026-01-14

# 搜索包含特定关键字的日志
grep "Redis" logs/app.log*

# 查看特定时间段的日志
awk '$1 == "2026-01-15" && $2 >= "10:00:00" && $2 <= "11:00:00"' logs/app.log
```

### 5.3 日志分析工具
可以使用以下工具进行日志分析：
- **ELK Stack**: Elasticsearch、Logstash、Kibana
- **Graylog**: 集中式日志管理平台
- **Splunk**: 企业级日志分析平台

## 6. 故障排查指南

### 6.1 常见日志问题
- **日志文件未生成**: 检查日志目录权限、磁盘空间
- **日志级别不生效**: 检查配置文件和环境变量设置
- **日志写入性能问题**: 确保使用了异步写入模式，检查磁盘I/O性能
- **日志文件过大**: 调整 `LOG_MAX_BYTES` 和 `LOG_BACKUP_COUNT` 参数

### 6.2 日志排查步骤
1. **查看错误日志**: 首先检查 `error.log` 文件，了解系统是否有错误
2. **按时间筛选**: 根据问题发生的时间，查看对应时间段的日志
3. **按模块筛选**: 定位问题所在的模块，查看该模块的日志
4. **按请求ID筛选**: 如果启用了请求ID，可以通过请求ID追踪完整的调用链
5. **分析日志模式**: 寻找重复出现的错误模式，识别问题的根本原因

## 7. 最佳实践

### 7.1 日志记录建议
- **合理使用日志级别**: DEBUG 用于详细调试信息，INFO 用于一般信息，WARN 用于警告，ERROR 用于错误
- **包含足够的上下文**: 日志内容应包含足够的上下文信息，便于问题排查
- **避免敏感信息**: 不要在日志中记录密码、令牌等敏感信息
- **结构化日志**: 对于复杂的日志信息，考虑使用结构化格式（如 JSON）
- **定期清理**: 确保日志清理机制正常工作，避免磁盘空间耗尽

### 7.2 性能优化建议
- **生产环境使用 INFO 或 WARN 级别**: 减少日志量，提高系统性能
- **合理设置缓冲区大小**: 根据系统特点调整缓冲区大小
- **使用异步写入**: 确保启用了异步日志写入
- **避免频繁的 DEBUG 日志**: 在性能敏感的代码路径中，避免过多的 DEBUG 级日志

## 8. 总结

本日志配置方案实现了以下目标：
- **完整的日志级别控制**: 支持 ERROR、WARN、INFO、DEBUG 四个级别
- **规范的日志格式**: 包含时间戳、日志级别、模块名称、请求ID、具体日志内容
- **系统化的日志存储**: 统一存储至指定文件夹，按日期或文件大小进行轮转切割
- **高效的日志处理**: 采用异步写入方式，减少对系统性能的影响
- **自动化的日志管理**: 实现日志文件的归档与清理机制，保留指定天数的日志记录
- **便捷的日志查看**: 提供了多种日志查看方法和故障排查指南

通过这套日志配置方案，可以有效提高系统的可观测性，便于问题排查和系统监控。
